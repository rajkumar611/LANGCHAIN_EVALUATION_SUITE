DATASYNC PRO — TECHNICAL REFERENCE & TROUBLESHOOTING GUIDE
===========================================================

PRODUCT OVERVIEW
----------------
DataSync Pro is a real-time data integration platform that synchronises records between source databases and downstream systems. It supports event-driven pipelines, scheduled batch transfers, and webhook-triggered syncs. All operations are logged with structured audit trails.

The platform is organised into three processing layers: the ingestion layer (inbound connectors), the transformation layer (mapping rules and validation), and the delivery layer (outbound connectors and retry logic).

CORE CONCEPTS — DATA CONSISTENCY
----------------------------------
Data consistency in DataSync Pro is maintained through a two-phase commit protocol. When a sync job is initiated, the platform acquires a read lock on the source table, captures a point-in-time snapshot, and only releases the lock once the snapshot has been confirmed as fully transferred to the destination.

If the destination acknowledges receipt of all records within the configured timeout window (default: 30 seconds), the transaction is committed and the source lock is released. If the destination fails to acknowledge within the timeout, the transaction is rolled back and the source data remains unchanged.

This design ensures that no partial writes are ever delivered to the destination — a condition known as write atomicity. DataSync Pro enforces write atomicity at the record batch level, not at the individual record level.

CONNECTOR REFERENCE — INBOUND
-------------------------------
Connector Type: PostgreSQL Source
  Default batch size: 500 rows
  Read isolation: REPEATABLE READ
  Supported triggers: cron schedule, manual, webhook

Connector Type: MySQL Source
  Default batch size: 500 rows
  Read isolation: SNAPSHOT
  Supported triggers: cron schedule, manual, webhook

Connector Type: REST API Source
  Default batch size: 100 records (per page)
  Pagination: cursor-based
  Supported triggers: webhook, manual

CONNECTOR REFERENCE — OUTBOUND
--------------------------------
Connector Type: BigQuery Destination
  Write mode: APPEND or TRUNCATE_LOAD
  Streaming insert supported: yes (for batches under 10 MB)

Connector Type: Snowflake Destination
  Write mode: MERGE or APPEND
  Staging area: internal S3 bucket

Connector Type: Webhook Destination
  Payload format: JSON
  Retry policy: 3 attempts with exponential backoff

ERROR CODES AND RESOLUTIONS
-----------------------------

ERR-100: SOURCE_CONNECTION_TIMEOUT
  Severity: Medium
  Meaning: The platform could not establish a connection to the source database within the configured timeout period (default: 10 seconds). This is commonly caused by network latency spikes, source database overload, or firewall rule changes.
  Resolution: Check source database availability, verify network path, and increase the connection timeout via the connector settings if latency is consistently high.

ERR-200: SCHEMA_MISMATCH
  Severity: High
  Meaning: The column schema detected in the source table does not match the schema definition stored in the DataSync Pro mapping configuration. This error is triggered when a column is added, removed, renamed, or changes data type in the source.
  Resolution: Navigate to Mappings > Edit Mapping for the affected connector. Update the schema definition to reflect the current source structure. Re-run schema validation before restarting the sync job.

ERR-301: WRITE_ATOMICITY_VIOLATION
  Severity: Critical
  Meaning: The delivery layer detected that a partial batch was written to the destination before an interruption occurred. This violates write atomicity — the guarantee that either all records in a batch are written or none are. ERR-301 is triggered when the destination connector reports a partial acknowledgement rather than a full commit or full rollback.
  Resolution: Immediately pause all sync jobs using the affected destination connector. Run the Consistency Audit Tool (Admin > Tools > Consistency Audit) to identify records that were partially written. Manually reconcile destination records against the last confirmed full-batch checkpoint. Do not restart the sync job until reconciliation is complete. Partial writes to BigQuery can be reversed using the table snapshot feature if a snapshot was taken within 7 days.

ERR-401: AUTH_TOKEN_EXPIRED
  Severity: Low
  Meaning: The OAuth token used to authenticate the connector has expired. This is expected behaviour when long-running sync jobs exceed the token validity window without automatic refresh.
  Resolution: Rotate the connector credentials via Settings > Connectors > Credentials. DataSync Pro will automatically re-authenticate on the next sync cycle. For jobs that regularly exceed 60 minutes, enable the token auto-refresh option in connector settings.

ERR-503: DESTINATION_CAPACITY_EXCEEDED
  Severity: High
  Meaning: The destination system has rejected inbound data because it has reached its write capacity limit. For BigQuery streaming inserts, this typically occurs when the platform exceeds 100,000 rows per second per table. For Snowflake, it occurs when the virtual warehouse compute credits are exhausted.
  Resolution: Reduce the batch size in the destination connector configuration. For BigQuery, switch from streaming insert mode to batch load mode if sustained high-volume transfers are required. For Snowflake, upgrade the virtual warehouse size or adjust the credit allocation in your Snowflake account. ERR-503 does not cause data loss — the batch is queued and retried automatically every 5 minutes for up to 6 retry cycles.

ERR-720: TRANSFORMATION_RULE_FAILED
  Severity: High
  Meaning: A record failed validation against one or more transformation rules defined in the mapping configuration. ERR-720 is triggered when a mandatory field is null, a value falls outside a configured acceptable range, or a type conversion fails.
  Resolution: Review the transformation log (Logs > Transformation) to identify which field and rule caused the failure. Either fix the source data or update the transformation rule to handle the observed value. Affected records are moved to the Dead Letter Queue (DLQ) and held for 14 days.

PERFORMANCE TUNING
-------------------
For high-volume pipelines transferring more than 1 million rows per job, the following settings are recommended:

  Batch size: 1,000–5,000 rows (depending on row width)
  Parallelism: 4–8 worker threads
  Read isolation: SNAPSHOT (preferred over REPEATABLE READ for reduced lock contention)
  Write mode: BATCH LOAD (preferred over streaming for large volumes)

Enabling parallelism splits the source table into partitions, assigning each partition to an independent worker thread. Partition boundaries are computed by primary key range or by timestamp column depending on the connector configuration.

MONITORING AND ALERTS
----------------------
DataSync Pro emits structured metrics for every sync job. The following metrics are available in the monitoring dashboard:

  rows_read: Total records read from the source in the current job
  rows_written: Total records successfully written to the destination
  rows_failed: Total records moved to the Dead Letter Queue
  job_latency_ms: End-to-end time from job trigger to destination acknowledgement
  retry_count: Number of retry cycles executed in the current job

Alert thresholds can be configured per metric per pipeline. Alerts are delivered via email, Slack webhook, or PagerDuty integration.

AUDIT AND COMPLIANCE
---------------------
Every sync operation is recorded in the immutable audit log with the following fields: job_id, connector_id, trigger_type, start_time, end_time, rows_read, rows_written, rows_failed, error_code (if applicable), operator_id. Audit logs are retained for 90 days and can be exported in JSON or CSV format for compliance purposes.
