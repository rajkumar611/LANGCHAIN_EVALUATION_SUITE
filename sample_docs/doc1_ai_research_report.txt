1. EXECUTIVE SUMMARY
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Project Athena was initiated in Q2 2024 to address a critical gap in NeuralCore's enterprise AI offerings: the inability of standalone large language models to accurately answer questions grounded in proprietary, frequently updated knowledge bases. The project produced a production-grade RAG system named Athena-v2, which achieved a 91.4% faithfulness score on internal benchmarks, a 38% reduction in hallucination rate compared to the baseline GPT-4 deployment, and a mean latency of 1.2 seconds per query at the 95th percentile.

This report details the system architecture, experimental findings, benchmark results, known limitations, and the roadmap for Athena-v3.

2. BACKGROUND AND MOTIVATION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Prior to Project Athena, NeuralCore's enterprise clients relied on a naive prompt-stuffing approach: entire documents were concatenated into the LLM context window. This approach suffered from three critical problems.

First, context window limits meant that documents longer than approximately 32,000 tokens were silently truncated, causing the model to miss information that appeared later in long documents. A post-mortem on the Verizon Capital contract incident (September 2023) traced a $2.4M billing error to exactly this truncation — the pricing amendment was in the final section of a 90-page contract.

Second, the signal-to-noise ratio degraded severely as document length increased. Even when truncation did not occur, the model's attention mechanism diluted focus across irrelevant context, reducing answer precision by an estimated 22% on internal QA benchmarks.

Third, there was no mechanism to cite sources. Clients in regulated industries (finance, healthcare, legal) require that every AI-generated answer be traceable to a specific document passage for audit compliance.

These three problems collectively motivated the RAG architecture. By retrieving only the most relevant document chunks at query time, Athena-v2 solves truncation, improves signal-to-noise, and enables chunk-level citation.

3. SYSTEM ARCHITECTURE
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Athena-v2 is composed of five sequential stages: ingestion, indexing, retrieval, reranking, and generation.

3.1 Ingestion Pipeline

Documents are accepted in PDF, DOCX, and TXT formats via a REST API endpoint. The ingestion service extracts raw text, normalises Unicode characters, strips boilerplate headers and footers using a trained classifier (98.2% accuracy on internal test set), and splits the cleaned text into overlapping chunks of 400 characters with a 50-character overlap. Overlap ensures that sentences straddling chunk boundaries are not split mid-context. Chunks shorter than 40 characters — typically section headings or lone page numbers — are discarded as noise.

3.2 Indexing Layer

Each chunk is embedded using the all-MiniLM-L6-v2 sentence transformer, producing a 384-dimensional dense vector. These vectors are stored in a FAISS flat index for exact cosine similarity search. In parallel, chunks are indexed in a TF-IDF sparse matrix using scikit-learn's TfidfVectorizer with English stop-word removal. Maintaining both indexes enables hybrid retrieval in Stage 3. The dual-index build completes in under 800 milliseconds for a 300-chunk corpus on a single CPU core, making real-time re-indexing after document upload feasible.

3.3 Retrieval

The retrieval stage implements three strategies depending on the query classifier's output. For factual lookup queries (e.g., "What is the cancellation policy?"), BM25 sparse retrieval is favoured because exact keyword matching outperforms semantic similarity for terminology-heavy questions. For conceptual queries (e.g., "Explain the approach to risk management"), dense vector retrieval is favoured because paraphrased semantics matter more than exact terms. For mixed queries, Reciprocal Rank Fusion (RRF) with k=60 combines both ranked lists into a single fused ranking. The k=60 constant was selected after a grid search over k ∈ {10, 30, 60, 100} on the internal QA benchmark; k=60 yielded the highest NDCG@10 score of 0.847.

3.4 Reranking

The top-20 chunks from retrieval are passed to a cross-encoder reranker (ms-marco-MiniLM-L-6-v2) that scores each chunk against the original query as a pair. The reranker produces a relevance score between 0 and 1. Chunks scoring below 0.35 are discarded. The remaining chunks, sorted by reranker score, are passed to the generation stage. Reranking adds approximately 180 milliseconds of latency but reduces irrelevant context by 41% on average.

3.5 Generation

The final generation prompt concatenates the top-k reranked chunks (k ≤ 8 by default) with the original user question and a system instruction enforcing citation and grounded-only responses. The LLM is instructed to respond with "I don't have enough information" when retrieved context is insufficient, rather than hallucinating. This instruction alone reduced hallucination rate from 14.2% to 6.8% on the Athena evaluation benchmark (AEB-v2).

4. BENCHMARK RESULTS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Athena-v2 was evaluated on AEB-v2, a curated benchmark of 1,200 question-answer pairs across 40 enterprise document types. The evaluation framework measures four dimensions.

Faithfulness (answer claims supported by retrieved context): 91.4%
Answer Relevancy (answer addresses the question): 88.7%
Context Utilisation (relevant context chunks actually used): 84.2%
Hallucination Rate (claims not found in any document): 6.8%

Compared to the previous system (Athena-v1, which used naive top-k retrieval without reranking), Athena-v2 improved faithfulness by 12.3 percentage points and reduced hallucination by 38% (from 11.0% to 6.8%).

Latency at the 95th percentile was 1.2 seconds for a 300-chunk corpus. Latency scales sub-linearly with corpus size due to FAISS's approximate nearest-neighbour optimisations; a 3,000-chunk corpus increases P95 latency to only 1.7 seconds.

5. KNOWN LIMITATIONS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Despite the improvements in Athena-v2, three limitations remain unresolved.

The first limitation is multi-hop reasoning failure. When an answer requires combining information from two or more non-adjacent document sections, the retrieval stage may not surface both relevant chunks simultaneously. In AEB-v2, 18% of failures were traced to this root cause. For example, a question asking "What is the penalty if a Tier 3 client misses both their Q2 and Q4 payment deadlines?" requires retrieving the client tier definitions (Section 4.2 of the contract), the payment deadline schedule (Section 7.1), and the penalty clause (Section 9.4) — three separate retrievals that the current single-pass retrieval cannot perform.

The second limitation is table and figure blindness. The ingestion pipeline converts PDFs to plain text, which discards table structure. Numerical data in tables is often extracted as space-separated tokens that lose their row-column relationships. This causes retrieval to return garbled numeric context for financial and scientific documents.

The third limitation is language coverage. The all-MiniLM-L6-v2 embedder is optimised for English. Multilingual documents or code-switched text (common in Singapore and Indian enterprise clients) produce degraded embedding quality, reducing retrieval precision by approximately 31% on the internal multilingual test set.

6. ROADMAP: ATHENA-V3
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Athena-v3 is scheduled for internal release in Q3 2025. The three primary objectives map directly to the limitations identified above.

To address multi-hop reasoning, Athena-v3 will implement an agentic retrieval loop. Rather than a single retrieval pass, the system will iteratively issue search queries, inspect retrieved results, and decide whether additional searches are needed to answer the question. This loop is bounded at a maximum of four iterations to prevent runaway latency. Early experiments show a 29% reduction in multi-hop failure rate on a held-out test set.

To address table blindness, Athena-v3 will integrate a table extraction module (based on Microsoft Table Transformer) that converts table cells into structured JSON before chunking, preserving row and column relationships. Retrieved table chunks will be rendered as markdown tables in the generation prompt.

To address language coverage, the all-MiniLM-L6-v2 embedder will be replaced with paraphrase-multilingual-MiniLM-L12-v2, which supports 50 languages. Preliminary benchmarks show parity with the English-only model on English queries while recovering 24 of the 31 percentage points lost on multilingual queries.

7. CONCLUSION
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Project Athena demonstrates that a well-engineered RAG pipeline substantially outperforms naive LLM deployments on enterprise document QA tasks. Athena-v2's 91.4% faithfulness score and 6.8% hallucination rate represent meaningful improvements over both the naive baseline and Athena-v1. The three unresolved limitations — multi-hop reasoning, table blindness, and multilingual coverage — are directly addressed in the Athena-v3 roadmap. The team recommends promoting Athena-v2 to general availability for English-language enterprise clients while Athena-v3 development continues in parallel.
