What is RAGAS?
RAGAS (Retrieval Augmented Generation Assessment) is an open-source Python framework for automated evaluation of RAG pipelines. Published in 2023 by explodinggradients, it addresses a critical gap: traditional NLP metrics like BLEU or ROUGE measure surface-level word overlap, which is a poor fit for RAG outputs where a correct answer can be phrased in many ways.

RAGAS takes a different approach — it uses an LLM as a judge to evaluate answers, context, and retrieval quality using carefully designed prompts and scoring algorithms.

Most RAGAS metrics are reference-free, meaning they do not need a pre-written correct answer to compare against. Consider a customer support chatbot that handles 10,000 questions a day — nobody has written a verified ideal answer for each of those questions, and doing so would be expensive and impractical. Reference-free metrics like Faithfulness and Answer Relevancy can still evaluate quality just from the question, the generated answer, and the retrieved chunks — no human-curated answer required. Metrics that do require a reference answer (Context Precision, Answer Correctness) are reserved for offline testing where you have prepared a benchmark dataset.
Open-Source
MIT-licensed Python library on GitHub. Install with pip install ragas.
Reference-Free
Faithfulness and Answer Relevancy require no ground truth — usable on live production traffic.
LLM-as-Judge
Uses any LLM (Claude, GPT-4, Gemini) as the evaluator. RAGAS needs the LLM to return structured data (e.g. a list of claims, a relevance label) rather than free text — instructor is a Python library that guarantees this by forcing the LLM to respond in a specific JSON schema every time.
How RAGAS Evaluates
RAGAS does not produce a single overall score — it measures three distinct failure modes independently. Each metric targets a different part of the RAG pipeline that can break silently.

Metrics — How Each One is Calculated
ALWAYS SCORED
Faithfulness
What it measures: Whether every claim in the generated answer is actually supported by the retrieved context. It catches hallucinations — statements the LLM invented that cannot be traced back to the provided chunks.
How it is calculated
Step 1 — The judge LLM reads the answer and extracts every individual factual claim as a list. Example: "RAG uses vector search", "transformers were invented in 2017".
Step 2 — For each claim, the LLM checks whether it can be inferred from the retrieved context chunks. Returns a binary verdict per claim: supported (1) or not supported (0).
Step 3 — Score is computed as the ratio of supported claims to total claims.
Formula
Faithfulness = | supported claims | | total claims |
Score range: 0.0 (all claims hallucinated) → 1.0 (every claim is grounded)
High score (→ 1.0)
Every claim the answer makes can be traced to the provided context. No hallucination.
Low score (→ 0.0)
The LLM added facts from its training memory that are not in the retrieved chunks.
ALWAYS SCORED
Answer Relevancy
What it measures: Whether the generated answer actually addresses what the user asked. An answer can be factually correct and grounded in context but still be evasive, vague, or answer a different question than the one asked.
How it is calculated
Step 1 — The judge LLM reads the answer (not the original question) and generates N hypothetical questions that this answer could plausibly be responding to. Default N = 3.
Step 2 — Each generated question is embedded using the embeddings model (all-MiniLM-L6-v2 in this app).
Step 3 — Cosine similarity is computed between each generated question embedding and the original question embedding. The mean similarity is the score.
Formula
AnswerRelevancy = 1 N Σ cos(qgenerated_i, qoriginal)
Original question: "What are the side effects of ibuprofen?"
Generated Q1: "What are the risks associated with taking ibuprofen?" → cos = 0.94
Generated Q2: "What adverse effects does ibuprofen cause?" → cos = 0.91
Generated Q3: "Is ibuprofen safe to take regularly?" → cos = 0.72
Answer Relevancy = (0.94 + 0.91 + 0.72) / 3 = 0.857
Why generate questions instead of scoring directly?
Asking an LLM "is this answer relevant?" produces unreliable binary yes/no responses. Generating questions and measuring embedding distance gives a continuous, stable score that correlates better with human judgment.
NEEDS GROUND TRUTH
Context Precision
What it measures: Whether the retriever ranked the most useful chunks at the top. A retriever might fetch the right information but bury it at position 8 out of 10 — the LLM may never use it. Context Precision rewards having relevant chunks at the top of the retrieved list.
How it is calculated
Step 1 — For each retrieved chunk at rank k, the judge LLM is asked: "Given the question and the ground truth, is this chunk useful for answering the question?" Returns a binary relevance label (1 = yes, 0 = no).
Step 2 — Precision@k is computed for each rank position: how many of the top-k chunks are relevant?
Step 3 — Only positions with a relevant chunk contribute to the final average, weighted by their relevance label. Irrelevant chunks at early positions are penalised.
Example — 4 chunks retrieved, 2 relevant (at positions 1 and 3)
RankChunkRelevant?Precision@kContributes?
1chunk A✓ 11/1 = 1.0yes
2chunk B✗ 01/2 = 0.5no
3chunk C✓ 12/3 = 0.67yes
4chunk D✗ 02/4 = 0.5no
Context Precision = (1.0 × 1 + 0.67 × 1) / 2 = 0.833
↑ Only positions with rel=1 count. Chunk A at rank 1 boosts the score; chunk C at rank 3 reduces it slightly.
Why ground truth is required
Without a reference answer, RAGAS cannot determine which chunks are truly relevant vs irrelevant. The ground truth is used to give the LLM a benchmark to compare each chunk against.
NEEDS GROUND TRUTH
Answer Correctness
What it measures: How factually accurate the generated answer is compared to a known-correct reference answer. Unlike Faithfulness (which checks against retrieved context), Correctness checks against ground truth — it catches cases where the context was right but the LLM still produced a wrong answer.
How it is calculated
Step 1 — The judge LLM extracts individual factual statements from both the generated answer and the ground truth answer.
Step 2 — Each statement is classified into one of three buckets:
  TP (True Positive) — present in both generated answer and ground truth
  FP (False Positive) — present in generated answer only (incorrect or extra claim)
  FN (False Negative) — present in ground truth only (missed information)
Step 3 — An F1 score is computed from TP/FP/FN, then combined with a semantic similarity score (embedding cosine similarity between the two answers) using a weighted sum.
Formula
F1 = 2 × TP 2TP + FP + FN     AnswerCorrectness = w1 × F1 + w2 × SemanticSimilarity
Ground truth: "RAG retrieves relevant chunks and injects them into the prompt."
Generated: "RAG retrieves relevant chunks and uses a vector database."

TP: "RAG retrieves relevant chunks"
FP: "uses a vector database" (not in ground truth)
FN: "injects them into the prompt" (missed)

F1 = 2×1 / (2×1 + 1 + 1) = 0.5

Metric Quick Reference
Faithfulness
supported claims / total claims
Checks every claim in the answer against the retrieved context. Catches hallucinations — information the LLM added from training memory rather than the document.
Answer Relevancy
mean cosine sim of generated questions vs original
Generates N questions the answer could be responding to, then measures how close they are semantically to the actual question asked.
Context Precision (needs ground truth)
weighted mean Precision@K over relevant chunks
Measures whether relevant chunks are ranked first. Penalises retrieval systems that find the right chunk but rank it behind irrelevant ones.
Answer Correctness (needs ground truth)
weighted F1 + semantic similarity vs reference
Compares factual statements in the answer vs ground truth using TP/FP/FN classification, combined with embedding-based semantic similarity.
